# Learn — 学習ドキュメント自動生成

バージョンの変更内容から学習ドキュメントを自動生成する。2 種類のドキュメントを生成可能。

```
/learn v8          → 日報的な学習記録を生成
/learn v8 lesson   → Q&A 形式の授業文書を生成
/learn v8 both     → 両方を生成
/learn             → 最新バージョンを自動検出（学習記録を生成）
```

All output must be in Japanese.

## 生成ドキュメントの種類

| 種類 | ファイル名 | 対象読者 | 形式 |
|------|-----------|---------|------|
| 学習記録 | `YYYY-MM-DD_vN_learning.md` | 自分自身（振り返り用） | トピック別の概要・コード例・教訓 |
| 授業文書 | `vN_lesson.md` | チームメンバー・後から学ぶ人 | 先生と生徒の Q&A 対話形式 |

---

## 種類 1: 学習記録（デフォルト）

### 生成手順

#### 1. 変更内容の収集
- `git log` で対象バージョンのコミットを特定（コミットメッセージの `vN:` プレフィックス）
- `git diff` で変更ファイル一覧と差分を取得
- `docs/01_requirements/requirements_vN.md` から学習テーマを抽出
- `docs/02_design/architecture_vN.md` から設計判断を抽出

#### 2. 分析観点
以下の観点で変更を分析し、学びとして抽出する:

| 観点 | 内容 |
|------|------|
| **新しい AWS サービス** | 初めて使ったサービスの概要と設定ポイント |
| **新しいツール・設定** | Claude Code hooks, skills, agents 等の設定パターン |
| **設計判断** | なぜその方式を選んだか（代替案との比較） |
| **直面した課題** | エラー、予想外の挙動、ハマりポイント |
| **Terraform パターン** | 新しく学んだ HCL のイディオム |
| **CI/CD の変更** | パイプラインの拡張ポイント |
| **セキュリティ** | セキュリティ関連の学び |

#### 3. 出力フォーマット

`docs/06_learning/YYYY-MM-DD_vN_learning.md` に以下の構造で生成:

```markdown
# 本日の学び (YYYY-MM-DD) — vN 実装を通じて

| 項目 | 内容 |
|------|------|
| 日付 | YYYY-MM-DD |
| 学習者 | m-horiuchi |
| テーマ | （requirements から抽出） |

---

## 1. トピックタイトル
### 概要
### コード例
### 教訓

## 2. ...

---

## まとめ — vN で学んだこと
### カテゴリ別要約
```

## 種類 2: 授業文書（lesson）

### 生成手順

#### 1. 情報収集
- `docs/01_requirements/requirements_vN.md` から変更概要・学習目標・機能要件を取得
- `docs/02_design/architecture_vN.md` から設計判断とアーキテクチャ図を取得
- `docs/02_design/infrastructure_vN.md` からリソース一覧を取得
- 実装コード（`app/`, `infra/`, `.claude/` 等）から具体的なコード例を取得
- 過去の授業文書（`docs/06_learning/v*_lesson.md`）からフォーマットを参照

#### 2. 授業文書の構造

先生と生徒の Q&A 対話形式で、以下の構造に従う:

```markdown
# vN 授業: {テーマタイトル}

---

## 先生、vN って何をやったんですか？

{テーマの概要を 2-3 文で説明}

大きく **N つ** のことをやった。

| # | やったこと | 一言で言うと |
|---|-----------|------------|
| 1 | ... | ... |
| 2 | ... | ... |

---

## 1. {トピック 1} って何ですか？

### 先生、前のバージョンまではどうなってたんですか？

{vN-1 までの状態と課題を説明}

### それの何が足りないんですか？

{課題を具体的に列挙}

### {トピック 1} を入れるとどうなるんですか？

{解決策と効果を説明。ASCII 図やコードブロックで視覚的に}

---

## 2. {トピック 2} って何ですか？

（同様の Q&A パターンを繰り返す）

---

## まとめ: vN で学んだこと

| # | 学んだこと | 一言 |
|---|-----------|------|
| 1 | ... | ... |

**vN のキーメッセージ**: {このバージョンの核心を一文で}
```

#### 3. 授業文書の執筆ルール

| ルール | 詳細 |
|--------|------|
| **対話形式** | 生徒の「先生、〇〇って何ですか？」→ 先生の回答の繰り返し |
| **段階的説明** | 前バージョンの状態 → 課題 → 解決策 → 効果の順で説明 |
| **具体例優先** | 抽象的な説明より、コードブロック・設定例・図を多用 |
| **比較で理解** | 「v(N-1) まではこう → vN からはこう」のビフォーアフター |
| **短い段落** | 1 段落は 3 文以内。長くなったらサブ見出しで分割 |
| **一言まとめ** | 各トピック末尾に「一言で言うと」を付ける |
| **冒頭の概要表** | 全トピックを表形式で一覧化（読者が全体像を把握してから詳細に入れるように） |
| **末尾のまとめ表** | 学んだことを表形式で振り返り + キーメッセージ |

#### 4. 出力先

`docs/06_learning/vN_lesson.md` に生成。

---

## 生成後の共通手順
- ユーザーにレビューを依頼
- 修正・追記の要望に対応
- 最終版を `docs/06_learning/` に保存

## 注意事項
- `docs/06_learning/` は `.gitignore` に含まれるためコミット対象外
- 過去の学習ドキュメント（`docs/06_learning/` 内）のフォーマットに合わせる
- 機密情報（AWS ID、リソース ARN 等）は含めない
- 授業文書は過去の `v*_lesson.md` を必ず参照してトーンを統一する
