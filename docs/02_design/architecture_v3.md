## 変更概要

v2 のアーキテクチャに以下を追加する:
- ECS Service Auto Scaling（CPU 負荷ベースで 1〜3 タスクに自動調整）

- HTTPS 関連リソースの Terraform コード整備（未デプロイ）

## 4. 通信フロー

### 4.1 リクエスト処理フロー（v3）

```
User → Internet → ALB (:80) → ECS Task x1〜3 (:8000) → FastAPI → SQLAlchemy → RDS Primary (:5432)
```

- ALB が複数 ECS タスクにラウンドロビンで分散（Auto Scaling によりタスク数変動）
- タスクが増減しても ALB のターゲットグループ登録が自動更新されるため、ユーザー影響なし

### 4.2 Auto Scaling フロー

```
CloudWatch Alarm (CPU > 70%) → Application Auto Scaling → ECS Service desired_count +1
CloudWatch Alarm (CPU < 40%) → Application Auto Scaling → ECS Service desired_count -1
```

1. ECS タスクの CPU 使用率が 70% を超えると CloudWatch Alarm が発火
2. Application Auto Scaling が ECS サービスの `desired_count` を +1 する
3. 新しい Fargate タスクが起動し、ALB のターゲットグループに自動登録
4. CPU 使用率が下がると（target tracking が判断）スケールインが発生
5. クールダウン期間（スケールアウト 60 秒 / スケールイン 300 秒）が経過後、次のスケーリングを許可

> **設計判断:** Target Tracking Scaling Policy は CloudWatch Alarm を自動生成する。
> 手動でアラームを定義する必要はない。

### 4.3 RDS Multi-AZ フェイルオーバーフロー

```
Primary RDS (AZ-a) 障害
    → CloudWatch が検知
    → RDS が Standby (AZ-c) を Primary に昇格
    → DNS フェイルオーバー（約 60〜120 秒）
    → ECS タスクが再接続（SQLAlchemy 接続プール）
```

> **アプリへの影響:** フェイルオーバー中（約 60〜120 秒）は DB 接続エラーが発生する可能性がある。
> SQLAlchemy のデフォルト接続プール設定で自動再接続するため、アプリ側の変更は不要。

### 4.4 HTTPS フロー（コードのみ）

```
User → Route53 DNS → ALB (:443, HTTPS) → ECS Task (:8000)
                      ↑
              ACM 証明書（DNS 検証済み）
User → ALB (:80, HTTP) → 301 Redirect → https://...
```

### 4.5 デプロイフロー（変更なし）

```
git push → GitHub Actions → Build Image → Push to ECR → Update ECS Task Definition → Rolling Deploy
```

## 5. Auto Scaling 設計詳細

### 5.1 スケーリング設定

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| スケーラブルターゲット | ECS サービス | `aws_appautoscaling_target` |
| ポリシータイプ | Target Tracking | 目標値に向かって自動調整 |
| メトリクス | ECSServiceAverageCPUUtilization | ECS サービス全タスクの平均 CPU |
| ターゲット値 | 70.0 (%) | CPU が 70% になるようにタスク数を調整 |
| 最小タスク数 | 1 | コスト最小化のため 0 は設定しない |
| 最大タスク数 | 3 | 学習用の上限（本番では 10+ が一般的） |
| スケールアウト クールダウン | 60 秒 | 新タスク起動後、次のスケールアウトを待機 |
| スケールイン クールダウン | 300 秒 | 急激な縮小を防ぎ、スパイクへの再対応を可能に |

> **設計判断 - クールダウンの非対称設定:**
> スケールアウトは素早く（60秒）、スケールインはゆっくり（300秒）する。
> これはトラフィックスパイクへの迅速対応と、過剰なスケールイン・スケールアウトの繰り返し（フラッピング）防止のためのベストプラクティス。

### 5.2 スケーリング動作の例

| 状況 | CPU 使用率 | タスク数変化 |
|------|-----------|------------|
| 通常時 | 30〜60% | 1 タスク維持 |
| 負荷増加 | 70% 超 | 1 → 2 タスク |
| さらに負荷増加 | 70% 超（60秒後） | 2 → 3 タスク |
| 負荷減少 | 40% 以下 | 3 → 2 → 1 タスク（300秒ごと） |
