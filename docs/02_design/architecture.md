## 3. ネットワーク構成

### 3.1 VPC 設計

| 項目 | 値 |
|------|------|
| VPC CIDR | 10.0.0.0/16 |
| パブリックサブネット 1 | 10.0.1.0/24 (ap-northeast-1a) |
| パブリックサブネット 2 | 10.0.2.0/24 (ap-northeast-1c) |
| Internet Gateway | あり |
| NAT Gateway | なし（コスト削減） |

### 3.2 サブネット配置の設計判断

ECS タスクは **パブリックサブネット** に配置する。

**理由:**
- NAT Gateway は約 $45/月のコストが発生する
- 学習用プロジェクトのため、コスト最小化を優先（NFR-5）
- ECS タスクに Public IP を付与し、Internet Gateway 経由で ECR からイメージを Pull する
- セキュリティグループで不要なインバウンドを遮断することで安全性を確保

### 3.3 セキュリティグループ

**ALB 用セキュリティグループ:**

| ルール | プロトコル | ポート | ソース |
|--------|-----------|--------|--------|
| Inbound | TCP | 80 | 0.0.0.0/0 |
| Outbound | All | All | 0.0.0.0/0 |

**ECS タスク用セキュリティグループ:**

| ルール | プロトコル | ポート | ソース |
|--------|-----------|--------|--------|
| Inbound | TCP | 8000 | ALB セキュリティグループ |
| Outbound | All | All | 0.0.0.0/0 |

## 4. 通信フロー

### 4.1 リクエスト処理フロー

```
User → Internet → ALB (:80) → ECS Task (:8000) → FastAPI App → Response
```

1. ユーザーが ALB の DNS 名に HTTP リクエストを送信
2. ALB がリクエストを受信し、ターゲットグループの ECS タスクに転送
3. FastAPI アプリケーションがリクエストを処理しレスポンスを返却
4. ALB 経由でユーザーにレスポンスを返却

### 4.2 デプロイフロー

```
git push → GitHub Actions → Build Image → Push to ECR → Update ECS Task Definition → Rolling Deploy
```

1. 開発者が `main` ブランチにコードを push
2. GitHub Actions ワークフローが起動
3. CI ステップ: lint, test, Docker build
4. CD ステップ: ECR にイメージを push、ECS タスク定義を更新
5. ECS サービスがローリングデプロイを実行

### 4.3 ヘルスチェックフロー

```
ALB → GET /health (30s interval) → ECS Task → {"status": "healthy"}
```

- ALB ターゲットグループが 30 秒間隔で `/health` にリクエスト
- 200 OK を返せば healthy、それ以外は unhealthy と判定
- unhealthy なタスクは自動的に置き換えられる
